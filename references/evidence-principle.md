# 证据原则

`ppt-deep-search` 是洞察型 Skill：先理解来源证据，再形成判断，最后产出下游 PPT 制作 Skill 可直接消费的 `ppt_content_brief.md`。

本文件只规定两件事：

1. 如何按金字塔原理组织大逻辑。
2. 如何写 `页面标题 + 标题说明 + 分析总结`。

具体表达例子见 `references/evidence-examples.md`。

## 大逻辑

遵循金字塔原理组织 PPT 逻辑：

1. 结论先行：先给观众最该记住的判断，再展开理由。
2. 以上统下：上层判断必须概括下层内容；下层内容必须支撑上层判断。
3. 归类分组：同一层的理由必须属于同一类问题，不能把机制、收益、风险、动作混成一组。
4. 逻辑递进：同一组理由必须有顺序，可按重要性、因果、时间、结构或程度排列。

用于 PPT 时：

- `Summary` 是整份 PPT 的核心结论。
- 章节标题可以是简短标签，方便目录扫描。
- 章节说明承载章节判断，说明这一组页面如何支撑 `Summary`。
- 页面标题是支撑章节判断的单页判断，不能退化成主题标签。
- 分析总结是支撑页面标题的 2-3 条子逻辑；2 条经常已经足够。
- 正文证据支撑对应的分析总结。

```text
Summary
-> 章节标题 1 + 章节说明
   -> 页面判断 A
   -> 页面判断 B
-> 章节标题 2 + 章节说明
   -> 页面判断 C
   -> 页面判断 D
-> 章节标题 3 + 章节说明
   -> 页面判断 E
```

对应到一页里，也是同一个结构：

```text
页面判断
-> 标题说明
-> 分析总结 1
   -> 正文证据
-> 分析总结 2
   -> 正文证据
-> 分析总结 3（仅在确有第三条独立子逻辑时）
   -> 正文证据 / 边界
```

不要按资料出现顺序写，不要把“背景、方法、实验、结论”直接搬成 PPT 结构。章节可以用短标签，但章节说明必须说清判断；页面标题不能只写标签，必须给出页面级判断。

每一层都要能互相检验：

- 下层内容回答不了上层的“为什么”，说明支撑失败。
- 上层判断概括不了下层内容，说明判断太窄或下层跑题。
- 同层理由互相包含、互相重复，说明拆分失败。

证据优先嵌在这条链路里。不要先写漂亮判断，再回头找材料。先识别来源材料里的事实、数字、图表、实验、案例、矛盾和边界，再决定能形成多强的判断。

可见判断只能来自证据：

- 来源直接支持时，可以写成强判断。
- 多个事实推导时，可以写成推断，但推理链要能在正文或备注里讲清。
- 证据冲突时，要呈现分歧，不能只挑有利结论。
- 证据薄弱时，只能弱表达，不能写成标题级断言。
- 没有证据支撑的判断，不能进入 `页面标题`、`标题说明`、`分析总结`。

同一层的理由要尽量互不重叠。判断方法：如果两个页面或两条分析总结删掉 label 后意思可以互换，说明没有拆出不同子逻辑，要重写。

每页只放能支撑当前页面判断的最小必要证据。完整配置、长表格、source locator 和审计细节不要挤进可见表达字段。

## 可见内容边界

`ppt_content_brief.md` 只写最终听众应该看到或听到的内容。

`页面标题`、`标题说明`、`分析总结` 和 `正文内容` 只能放结论、证据、解释和必要边界；不要放制作意图、写作提醒、版式建议、素材缺口、审稿说明或后续加工建议。

如果一句话是在告诉作者或制作者“这一页怎么写、怎么画、还缺什么、后面怎么处理”，它不能进入 PPT brief。

## 标题与总结

最终 brief 的可见表达字段固定为：

```text
页面标题：顶层页面判断
标题说明：判断的范围、口径、机制、价值或限制
分析总结：
- 子逻辑 A：支撑标题与说明的第一条理由
- 子逻辑 B：支撑标题与说明的第二条理由
- 子逻辑 C：可选；只有确有第三条独立理由或关键边界时才写
```

`页面标题 + 标题说明 + 分析总结` 是一个小金字塔：

- `页面标题` 负责给出本页判断。
- `标题说明` 负责补足判断的范围、口径、机制、价值或限制。
- `分析总结` 负责拆出 2-3 条子逻辑，支撑标题和说明；不要为了凑三条而补弱理由。
- `正文内容` 必须沿用分析总结的 label，展开对应证据。

### 页面标题

标题必须有对象，并表达价值、机制、变化、冲突或关键数字之一。标题默认控制在 18 个中文宽度单位以内。

优先写：

- 对象 + 数字收益
- 对象 + 成本变化
- 对象 + 机制变化
- 对象 + 风险分层
- 对象 + 反直觉判断

不要写：

- 章节名：`背景`、`实验结果`、`下一步计划`
- 论文小标题：`研究现状`、`关键机制`
- 管理姿态：`值得复现，不宜直上`
- 空泛结论：`证据够强，收益仍需验证`
- 只有英文方法名或产品名

标题可以短，但不能悬空。单看标题至少要知道对象和判断方向。

### 标题说明

标题说明必须和页面标题成对看。它不是第二个标题，也不是口号；它负责让标题变得可检视。默认控制在 60 个中文宽度单位以内。

优先补四类信息：

- 范围：模型、系统、场景、批量、数据集、用户群。
- 口径：数字怎么来的，对照是什么，时间窗口是什么。
- 机制：收益、风险或变化是怎么发生的。
- 限制：结论在哪些条件下成立。

不要把 `仍需复测`、`有待验证`、`不宜直上` 当作默认收尾。除非这一页专门讲风险，否则标题说明优先解释价值来源、机制和证据口径。

不要把内部参数直接丢给观众。参数必须翻译成人话，例如：

- `top_k=20` -> `每次只保留前 20 个候选`
- `batch=1` -> `批量为 1`

### 分析总结

分析总结是标题和说明下面的子逻辑，不是正文摘要，也不是论文提纲。每条格式固定为：

```text
- 小标题：解释
```

label 建议 2-8 个字，解释句建议 15-35 个字。解释句必须补新信息，不能重复标题或标题说明。

分析总结默认写 2-3 条。只有 1 条支撑逻辑时，不要单独制造分析总结；这一页就是在论证 `页面标题 + 标题说明`，把支撑材料直接放进正文内容。

不要默认使用 `机制 / 证据 / 边界 / 建议`。这些 label 太通用，只能临时保底；生成后必须尝试替换成对象化 label。

按页面对象选择拆法：

- 实验收益页：关键指标 / 实验口径 / 适用条件。
- 成本节省页：浪费来源 / 节省机制 / 适用前提。
- 风险治理页：风险分档 / 触发条件 / 处理动作。
- 方法论页：反直觉判断 / 操作清单 / 兜底动作。
- 痛点页：现象 / 时间代价 / 金钱代价 / 风险外溢。
- 周报复盘页：核心指标 / 状态变化 / 阻塞项。

好的 label 一看就知道这页在讲什么，例如 `冷启动`、`钱被浪费`、`塞进窗口`、`回滚兜底`。如果一个 label 离开这页还能套到 100 页上，它就太机械。

分析总结必须共同支撑 `页面标题 + 标题说明`。如果某条总结横向发散，或者不能在正文内容里找到同名展开，就删掉或重写。

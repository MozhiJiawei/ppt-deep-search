# 证据表达例子

本文件只放 `页面标题`、`标题说明`、`分析总结` 的完整表达例子。原则见 `references/evidence-principle.md`。

示例不得复用 forward-test 主题，避免测试失真。不要使用 TiDAR、Aegaeon、R-CLA、KV cache、GPU pooling、RAG memory、RTX Spark、Agent PC 或长上下文记忆相关例子。

## 正例

每个例子都要把 `页面标题`、`标题说明`、`分析总结` 放在一起看。单看标题不够，标题和说明合起来才是可检视判断；分析总结是下面的子逻辑支撑。

```text
页面标题：FlashAttention 把瓶颈从算力拉回访存
标题说明：它不近似注意力结果，而是通过分块和重计算减少 HBM 读写
分析总结：
- 矩阵不落地：避免完整 materialize 注意力矩阵，减少中间结果搬运
- 分块计算：把注意力拆成适合片上存储的小块，提升内存访问效率
- 精确输出：收益来自 IO 组织变化，不是牺牲注意力计算精度
```

```text
页面标题：LoRA 让大模型微调变成小矩阵训练
标题说明：冻结主干权重，只训练低秩适配器，把参数更新限制在可插拔增量里
分析总结：
- 主干冻结：基础模型不被整体改写，训练和回滚成本更低
- 低秩增量：任务差异被压进小矩阵，减少需要更新的参数规模
```

```text
页面标题：DPO 把偏好学习压成一次分类目标
标题说明：不再先训练奖励模型再做强化学习，而是直接用偏好对优化策略
分析总结：
- 奖励旁路：跳过显式 reward model，减少一层训练与校准成本
- 偏好成对：训练信号来自 chosen/rejected 对比，而不是单条答案打分
```

```text
页面标题：SAM 把分割从类别任务改成提示任务
标题说明：用户给点、框或粗略区域，模型输出可交互修正的候选 mask
分析总结：
- 提示输入：分割目标由 prompt 指定，不局限于固定类别表
- 候选输出：一次提示可生成多个 mask，交给用户或下游规则选择
- 泛化前提：开放场景能力来自大规模视觉数据，专业领域仍要抽检
```

```text
页面标题：MapReduce 把分布式计算收进三步接口
标题说明：开发者只写 map 和 reduce，系统负责切分、shuffle、容错和调度
分析总结：
- 接口收敛：复杂集群执行被压成 map、shuffle、reduce 的固定形状
- 失败重跑：节点故障后重算局部任务，而不是让整条作业从头失败
- 场景适配：适合批处理和大规模离线数据，不适合低延迟交互查询
```

```text
页面标题：Raft 用领导者把共识路径讲清楚
标题说明：选主、日志复制、提交规则被拆成可理解的状态机流程
分析总结：
- 先选领导：多数派确认 leader 后，写入路径集中到一个入口
- 日志对齐：跟随者按 leader 日志补齐或回退，维持顺序一致
```

```text
页面标题：React Fiber 把一次渲染拆成可中断工作
标题说明：渲染过程被重写成可切片调度的任务单元，为优先级和并发让路
分析总结：
- 工作切片：长渲染被拆小，浏览器有机会先处理高优先级交互
- 优先级调度：不同更新不再天然同权，用户输入可以排到更前
- 体验目标：机制服务于响应性，不等于每个页面都会自动更快
```

```text
页面标题：三档风险，三种处理
标题说明：不是所有 AI 行为都同等危险；先分清可撤销和不可撤销，再谈流程
分析总结：
- 可撤销：写草稿、生成图这类任务，错了能撤回，可以放开跑
- 半可撤销：提 PR、改 staging 数据，错了要回滚，需要人工复核
- 不可撤销：发邮件、付款、删生产数据，必须硬卡和双人审
```

```text
页面标题：本周：付费转化率回到了 3.8%
标题说明：6 个发布，3 个实验收敛，1 个阻塞项升级；整体健康但仍有风险点
分析总结：
- 转化回升：新 onboarding checklist 带动付费转化反弹
- 实验收敛：3 个实验已能判断方向，减少继续消耗
- 延迟回退：推荐服务压测拖慢 p95，需要加缓存
```

```text
页面标题：Solo founders duct-tape 7+ tools to stay alive
标题说明：一个人经营业务，却要在账单、CRM、合同、税务之间来回搬运信息
分析总结：
- 工具割裂：Stripe、表格、合同、CRM 各管一段，彼此不说话
- 时间丢失：真正该卖东西的时间，被复制粘贴吃掉了
```

```text
页面标题：递归只看两件事：何时停，如何变小
标题说明：递归函数不是魔法；停止条件和缩小步骤说清楚，代码才不会无限跑
分析总结：
- 停止条件：先说清什么时候直接返回，避免无限递归
- 递归步骤：把问题缩小一圈，再交给同一个函数处理
```

```text
页面标题：Hermes 冷启动快，但首次配置会卡人
标题说明：三次平均看起来顺手；问题集中在 MCP 环境变量和新手接入
分析总结：
- 冷启动快：安装和首次运行足够轻，适合临时拉起来试任务
- 配置卡人：MCP 环境变量要手动处理，第一次接入容易断
```

## 反例

反例按 `Bad / Why bad / Rewrite pattern` 写。不要只展示坏例子，要说明坏在哪里，以及应该按什么方向改。

```text
Bad：
页面标题：实验结果
标题说明：这一页介绍实验表现。
分析总结：
- 背景：介绍研究背景
- 方法：介绍核心方法
- 结果：介绍实验结果

Why bad：
标题是主题标签；说明没有对象和口径；分析总结是论文提纲。

Rewrite pattern：
改成“对象 + 指标变化 + 实验口径”，再把分析总结拆成关键指标、机制、适用条件。
```

```text
Bad：
页面标题：证据够强，收益仍需本地验证
标题说明：需要结合本地情况判断。
分析总结：
- 价值：整体收益明显
- 边界：还不能替代本地指标
- 建议：后续继续验证

Why bad：
标题像决策姿态，但没有说对象和收益；说明只是废话收尾；分析总结没有证据锚点。

Rewrite pattern：
先写对象带来的具体收益，再在标题说明里补实验口径；如果要写边界，必须说清模型、场景、任务或流量条件。
```

```text
Bad：
页面标题：LoRA 冻结主干，训练低秩矩阵
标题说明：使用低秩分解降低可训练参数量。
分析总结：
- 方法：介绍低秩适配器
- 训练：介绍冻结主干
- 部署：介绍加载 adapter

Why bad：
标题只说做法，不说听众为什么要关心；分析总结像作者提纲，不是页面子逻辑。

Rewrite pattern：
把机制改成价值，例如“LoRA 让大模型微调变成小矩阵训练”；分析总结拆成训练成本、参数增量和部署切换。
```
